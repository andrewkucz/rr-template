## Stack

React Router v7 (framework mode) · tRPC v11 · TanStack React Query v5 · Better Auth · Drizzle ORM (PostgreSQL) · nuqs · shadcn/radix · Tailwind CSS v4 · Biome · TypeScript

## Commands

```bash
npm run dev           # start dev server
npm run build         # production build
npm run start         # serve the production build
npm run typecheck     # react-router typegen + tsc
npm run format        # biome format
npm run lint          # biome check
npm run lint:fix      # biome check --write
npm run check         # typecheck, then lint
npm run knip          # find unused files, exports, and deps

npm run db:generate   # generate drizzle migration files
npm run db:migrate    # run pending migrations
npm run db:studio     # open drizzle studio

npm run gen:auth      # regenerate app/db/auth.schema.ts from better-auth config
```

There are no tests.

## Architecture

```
app/
  root.tsx              # app shell; mounts NuqsAdapter, TRPCQueryClientProvider, auth, and UI providers
  routes.ts             # route config via `flatRoutes()`
  routes/               # file-based route modules (for example `_authenticated.tsx`, `api.trpc.$.ts`)
  features/<feature>/   # feature-scoped UI/data modules; server procedures live here when needed
  components/           # shared UI, layouts, and data-table building blocks
  hooks/                # reusable React hooks, including nuqs-backed table state
  context/              # theme, direction, layout, and search providers
  lib/
    trpc/               # tRPC setup (server, client, query client, provider, auth plugin)
    auth/               # Better Auth setup (server, browser, hooks, provider, server utils)
  db/
    index.ts            # drizzle db instance (DATABASE_URL)
    schema.ts           # app tables
    auth.schema.ts      # generated by `gen:auth`; do not edit manually
```

**Request flow:** Browser → React Router file route → loader/action (server) or component (client). Client data fetching goes through tRPC at `/api/trpc`, handled by `routes/api.trpc.$.ts`. Client-side URL query param state goes through `nuqs` library

## tRPC Conventions

**Root router** lives in `app/lib/trpc/router.ts`. To add a new feature's routes:
1. Create `app/features/<feature>/router.ts` with your procedures
2. Import and mount it in `app/lib/trpc/router.ts`
3. Keep feature-specific helpers like `schemas.ts`, `utils.ts`, and other supporting files inside that same feature folder instead of centralizing them elsewhere

**Procedure types** (from `@/lib/trpc/server`):
- `publicProcedure` — no auth required
- `authProcedure` — requires active session; throws `UNAUTHORIZED` if not signed in; injects `session` into `ctx`

**Client usage** in React components:
```ts
const trpc = useTRPC();
const { data } = useQuery(trpc.<router>.<procedure>.queryOptions());
const mutation = useMutation(trpc.<router>.<procedure>.mutationOptions());
```

`useTRPC` and `TRPCProvider`/`useTRPCClient` are exported from `@/lib/trpc/client`. The app-wide React Query and tRPC wiring lives in `app/lib/trpc/provider.tsx`.

## URL State Conventions

- `nuqs` is the standard library for URL query param state management in client components
- The React Router adapter is already mounted in `app/root.tsx` via `NuqsAdapter`
- Prefer `useQueryState` / `useQueryStates` with parser helpers such as `parseAsString`, `parseAsInteger`, `parseAsIndex`, and enum/array parsers
- Keep query param names explicit and stable with `urlKeys` when the state shape should not leak directly into the URL
- For table/list UIs, prefer colocating reusable query-state hooks in `app/hooks/` rather than duplicating parsing logic in each component

## Auth Conventions

**Server-side** (loaders/actions):
- `requireAuth(request)` — redirects to `/sign-in` if unauthenticated; returns `userId`
- `getUser(request)` — returns user object or `null`; never throws
- Both are in `@/lib/auth/server-utils`

**Client-side:**
- `authClient` from `@/lib/auth/browser`
- helper hooks exported from `@better-auth-ui/react` library (docs: https://better-auth-ui.com/llms.txt)

Auth tables are generated. After changing `app/lib/auth/server.ts`, run `npm run gen:auth`. Only generate and apply a migration when the user explicitly asks for schema changes to be committed.

## Database Conventions

- Drizzle instance exported from `@/db` as `db`
- App schema in `app/db/schema.ts`; import `user` from `./auth.schema` for foreign keys
- In handwritten Drizzle schemas, omit explicit column-name strings like `text("user_id")` or `timestamp("created_at")`; prefer `text()`/`timestamp()` so Drizzle defaults the SQL column name from the object key
- Always use Drizzle query builder; raw SQL only when strictly necessary
- When instructed to commit schema changes run `npm run db:generate` followed by `npm run db:migrate`. Do not do this unless instructed.

## Routing

Routes are discovered from `app/routes/` via file-based routing using `flatRoutes()` in `app/routes.ts`.

The route tree currently uses underscore-prefixed layout/group files such as `_authenticated.tsx` and nested settings routes like `_authenticated.settings.account.tsx`. Follow the existing file-route naming pattern when adding pages.

Follow React Router's file-route conventions: https://reactrouter.com/how-to/file-route-conventions

## Utilities & Aliases

- **Path alias**: `@/*` → `app/*` (e.g. `@/lib/utils`, `@/db`)
- **`cn()`** utility in `@/lib/utils` — combines `clsx` + `tailwind-merge`; use for all className concatenation
- shadcn components can be added with `shadcn` CLI
